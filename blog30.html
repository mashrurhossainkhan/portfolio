Jenkins Agent Lab

Connecting a Jenkins Agent Node on EC2

This lab shows how a Jenkins controller can use a separate EC2 instance as an agent node. The goal is to prepare the node, connect it from Jenkins, run a job on it, and confirm the result from the console output.

EC2 Agent → Security Group Access → Java Setup → Jenkins Node → Job Execution → Console Verification

Jenkins agents are useful because the controller does not have to run every build by itself. A separate agent can provide its own CPU, memory, tools, workspace, and operating system environment.

In this lab, the agent is an EC2 instance. Jenkins connects to it over the network, prepares a remote workspace, and sends the job there when the job is assigned to that node.

Main lesson: Jenkins agents make build work more scalable because the controller coordinates the job while the node performs the actual execution.

What This Setup Needs

Jenkins controller

The existing Jenkins server manages jobs, credentials, node configuration, and build history.

EC2 agent node

A separate EC2 instance is launched to work as the Jenkins build machine.

Network access

The agent security group must allow the Jenkins controller to reach it over the required SSH port.

Java runtime

Java is installed on the agent because Jenkins needs Java to run the remoting process.

Remote directory

A directory such as /opt or a dedicated Jenkins folder gives the agent a stable workspace location.

Jenkins job

A first test job confirms that Jenkins can schedule work on the new node successfully.

Step 1

Launch an EC2 Instance for the Agent

The lab starts by launching a new EC2 instance that will become the Jenkins agent. Keeping the agent separate from the controller makes the setup closer to a real CI/CD environment, where build workloads often run on dedicated machines.

Step 2

Allow the Jenkins Controller to Reach the Agent

The agent node must allow traffic from the Jenkins controller. In an EC2 setup, this usually means updating the agent security group so SSH access is allowed from the Jenkins server's private IP address or trusted source range.

Step 3

Install Java and Prepare the Agent Directory

After connecting to the EC2 instance, the next step is to install OpenJDK 21 and create a directory Jenkins can use for agent files and job workspaces. This gives Jenkins a known place to run commands and store temporary build data.

sudo apt update sudo apt install openjdk-21-jdk -y java -version sudo mkdir -p /opt/jenkins-agent sudo chown -R ubuntu:ubuntu /opt/jenkins-agent
Step 4

Add the EC2 Instance as a Jenkins Node

Inside Jenkins, the EC2 instance is added from the node management page. This is where the node name, remote root directory, labels, launch method, host, and credentials are connected together.

For this lab note, also copy the required .pem key code to the node and restrict its permissions. For example, place it in the node user's home directory and run chmod 400 your-key.pem so the key is not left with open permissions.
scp -i controller-key.pem agent-key.pem ubuntu@AGENT_PRIVATE_IP:/home/ubuntu/agent-key.pem ssh ubuntu@AGENT_PRIVATE_IP chmod 400 /home/ubuntu/agent-key.pem
Step 5

Run the First Job on the Agent

Once the node is connected, a first job is used to confirm that Jenkins can schedule work on the agent. A small test job is a good way to prove the connection before adding heavier build steps.

Step 6

Check the Console Output

The console output is the final proof that Jenkins actually ran the job. If the job was assigned correctly, the output should show the remote workspace and the commands executed on the agent node.

What I Learned

Agents need preparation

Jenkins can only use the node after the network, Java runtime, user permissions, and workspace path are ready.

Labels control placement

Labels help Jenkins decide which jobs should run on which agent, especially when different nodes have different tools.

Console logs matter

A successful console output is the clearest sign that the controller and agent are communicating properly.

A Jenkins agent setup is easiest to debug in this order: confirm SSH/network access, confirm Java, confirm the remote directory, confirm credentials, then run a small test job before using the node for a real build.
