Jenkins CI/CD Lab

What This Jenkins CI/CD Lab Actually Taught Me

This lab was not only about clicking through Jenkins screens. It helped me understand how a CI/CD tool connects source code, triggers, build steps, post-build actions, and final artifacts into a repeatable delivery workflow.

Source Code → Jenkins Job → Build Steps → Post-Build Actions → Artifact and Delivery Feedback

I wanted this lab to be more than a sequence of screenshots. The useful part was learning how Jenkins thinks: a job has a source, a trigger, a build process, and some action that happens after the build is done.

That mindset matters because CI/CD is not just automation. It is a structured way to reduce manual repetition and make software delivery more consistent.

Main lesson: Jenkins makes more sense when I stop seeing it as a dashboard and start seeing it as a delivery engine that moves code through a defined workflow.

Tools Used in This Lab

After reviewing the screenshots, these were the main tools and configurations visible in the Jenkins setup.

Jenkins

Jenkins was the CI/CD controller used to create jobs, manage build history, run build steps, and archive final artifacts.

Git and GitHub

The Java project source was connected through Git from the GitHub repository hkhcoder/vprofile-project.git.

JDK 17

The Java job was configured to use JDK17, with another JDK option also visible in the Jenkins dropdown.

Apache Maven 3.9

Jenkins used Maven as the build tool, with the build step set to Invoke top-level Maven targets and the goal install.

WAR Artifact

The build produced a deployable vprofile-v2.war file, which was then exposed through Jenkins build artifacts.

Artifact Archiving

The post-build action was configured to archive **/*.war, which made the build output easy to keep and review after each run.

Why This Lab Matters

In real projects, the problem is rarely just writing code. Teams also need a reliable way to fetch source code, run builds, create artifacts, and provide clear feedback when something succeeds or fails.

Job structure

Jenkins jobs are built from small pieces such as source code settings, triggers, build commands, and post-build actions.

Repeatability

Once a job is configured, the same process can run again and again without relying on memory or manual steps.

Visibility

Build history and console output make it much easier to understand what Jenkins did and where a problem happened.

Starting with the Jenkins Dashboard

The build history view immediately shows an important CI/CD idea: every execution leaves a record. That means a build is not just a one-time action. It becomes something traceable, reviewable, and comparable over time.

Step 1

Create a Freestyle Project

The lab began with a freestyle project. This is a useful starting point because it exposes the main Jenkins building blocks without requiring a full pipeline script.

Step 2

Add Purpose and Basic Project Information

Writing a project description may look minor, but it is part of making automation maintainable. Clear job names and descriptions help future users understand what the job is supposed to do.

Step 3

Configure the Build Environment

Jenkins cannot build Java projects correctly unless the tool configuration is ready. The JDK settings are a reminder that CI/CD still depends on real runtime requirements underneath the automation. In the screenshot, the Java project is set to use JDK17, and the Jenkins node label is MavenBuilder.

Step 4

Point Jenkins to the Source Code

A CI/CD job needs a clear source of truth. In Jenkins, the source code section is where the job learns what repository or code base it should pull from before doing any build work. The screenshot shows Git selected with a GitHub repository URL and the branch specifier set to */electron.

Step 5

Decide When the Job Should Run

Triggers are what turn a build job into part of a delivery system. Whether the build starts manually, on a schedule, or from a code event, Jenkins needs a rule for when to act.

Step 6

Add Build Steps and Post-Build Actions

This is where the job becomes a workflow. Build steps define what Jenkins should execute, and post-build actions define what should happen after that execution finishes.

Step 7

Understand the Build Action Section

The build action area shows the actual task Jenkins will run. In simple jobs, this may be a shell command. In larger jobs, it can represent a more complex build flow.

Step 8

Use Post-Build Actions for Follow-Up Work

Post-build actions are important because a successful build is often not the end of the story. Teams may archive artifacts, send notifications, or trigger the next stage after the build.

Step 9

Run the Freestyle Job and Review the Output

Triggering the build is where configuration turns into evidence. Once the job runs, Jenkins gives feedback through build status and console output.

Moving from a Simple Job to a Java Build

After the freestyle project, the lab moved into a more realistic Java build job. This stage connected Jenkins more directly to the kind of Maven-based project flow I had already seen in my earlier build lab.

Source code -> Jenkins -> Build tool -> Artifact -> Next delivery step
This was the point where Jenkins stopped feeling like a generic automation tool and started feeling like the coordinator for an actual software delivery process.
Step 10

Create the Java Maven Project in Jenkins

The Java project setup showed how Jenkins can be pointed at a real application workflow instead of a toy example. It is the same idea as the freestyle job, but now the build logic matters much more because it produces a real application artifact.

Step 11

Attach the Java Project Source

Source configuration matters even more in a real build because Jenkins now needs the correct project files, including the Maven structure that controls compilation and packaging.

Step 12

Set Triggers and Environment for the Java Build

This part reinforced the same lesson from the first job: automation still depends on clear start conditions and a known environment. Jenkins becomes dependable when those conditions are defined instead of assumed.

Step 13

Define the Maven Build Steps

This is where Jenkins becomes tightly connected to the actual build tool. Instead of running a random command, the job is now explicitly designed to execute the Maven-based build flow for the application. The screenshot shows Maven version 3.9 with the goal set to install.

Step 14

Configure Post-Build Actions for the Java Job

Post-build actions become especially useful once artifacts are involved. This is the place where Jenkins can archive outputs, publish reports, or prepare the result for the next stage in a deployment chain. In this lab, Jenkins was configured to archive **/*.war, which makes the generated package easy to preserve and download later.

Step 15

Run the Java Build and Check the Artifact

The final execution tied the whole lab together. Jenkins ran the configured build, and the output showed the result of that process as a generated WAR file.

Important distinction: Jenkins does not replace the build tool. It coordinates the process. Maven still handles the project build logic, while Jenkins manages when and how that build runs.

What the Lab Actually Taught Me

How CI is structured

A useful build job needs source settings, execution rules, a build action, and clear output.

Why visibility matters

Build history and console logs make automation easier to trust because I can inspect what happened.

Why artifacts matter

The generated WAR file is proof that the build pipeline created something deployable, not just a success message.

How CI and build tools connect

Jenkins coordinates the workflow, while Maven performs the Java project build logic underneath.

Why repeatability is powerful

Once the job is configured, the same process can run consistently instead of depending on manual memory.

Why this is DevOps foundation

CI/CD begins with understanding how code flows through automated steps toward a real deployment result.

My Biggest Takeaway

The most useful part of this Jenkins lab was realizing that automation is still built on understanding. If I do not understand the source, the build tool, the artifact, and the next delivery step, then clicking through Jenkins will not help much.

But once I understand that flow, Jenkins becomes much easier to reason about. It stops feeling like a complicated UI and starts feeling like a clear system for turning repeatable engineering steps into delivery automation.

This lab gave me a practical bridge from manual build work to automated delivery thinking, which is exactly where CI/CD becomes meaningful.

Jenkins Tools Setup

The screenshots at the end of the lab show another important idea: before a Jenkins job can use tools like Java, Maven, or Git, those tools are usually configured globally in Manage Jenkins. This is different from the job-level configuration.

Simple rule: first add tools in Jenkins global settings, then select those tools inside the job configuration.
Manage Jenkins -> Tools -> Add JDK / Maven / Git / NodeJS Job Configuration -> Select JDK / Maven version -> Use the tool in build steps

In other words, Jenkins jobs do not magically know where Java or Maven lives. The Jenkins administrator usually defines those installations once, and the jobs reuse them later.

Step 16

Open the Global Tools Area

From the Jenkins administration page, the Tools section is where global tool paths and automatic installers are managed. This is the starting point for adding build dependencies that jobs can reuse.

Step 17

Review the Maven Configuration Defaults

Jenkins can keep default Maven settings providers here. This is useful because Maven behavior can be centralized instead of being repeated manually in every project.

Step 18

Add JDK Installations

The JDK installations page shows named Java runtimes such as JDK17 and JDK21. Once these are saved globally, a job can simply choose the correct JDK by name.

Step 19

Add Git and Optional Gradle Tools

Jenkins can also define source-control and build tools globally. In this screenshot, Git is configured with the executable path set to git, and Gradle appears as another tool category that could be added for Gradle-based projects.

Step 20

Add Maven Installations

This screenshot shows Jenkins managing Maven versions such as Maven 3.9 and Maven 3.9.14, with automatic installation from Apache. This makes it easier to standardize the Maven version used by different jobs.

Step 21

See Other Tool Types Such as NodeJS

Jenkins supports more than Java-related tools. The NodeJS section shows that the same idea applies to other ecosystems: define the tool globally first, then let jobs use it later.

If you want to add tools in Jenkins, the path is usually Manage Jenkins → Tools. There you can add a JDK, Maven, Git, NodeJS, and other supported tools, then choose those named tools later inside each job.
