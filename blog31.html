Jenkins Plugins and Versioning

Jenkins Plugins, Variables, and Artifact Versioning

In this Jenkins lab, I practiced how plugins extend Jenkins, how build variables make jobs dynamic, and how versioned artifacts make each build easier to trace, deploy, and roll back.

Plugins → Environment Variables → Parameterized Builds → Timestamp Values → Versioned WAR Artifacts

This practice started with a simple problem: a Maven build creates the same WAR file name each time. That works for a basic build, but it is not enough when every CI/CD run needs a clear artifact that can be identified later.

The goal was to keep the Jenkins job simple while making the output more useful. Instead of only building target/vprofile-v2.war, the job copies that file into a versions folder with a unique name.

Main lesson: a Jenkins build should not only produce an artifact. It should produce an artifact with enough identity to support deployment, rollback, debugging, and release management.

What I Practiced

Plugin manager

I reviewed where Jenkins plugins are installed, updated, and managed from the Jenkins dashboard.

S3 publisher

I looked at how an S3 plugin can extend Jenkins so build artifacts can be published outside the local workspace.

Build variables

I used Jenkins variables such as $BUILD_ID, $VERSION, and $BUILD_TIMESTAMP.

Versioned artifacts

I copied the generated WAR file into a separate folder using a name that changes on each build.

Step 1

Open the Jenkins Plugin Manager

Jenkins plugins are extensions that add features to Jenkins. The plugin manager is available from Dashboard > Manage Jenkins > Plugins. From there, Jenkins shows available updates, available plugins, installed plugins, and advanced settings.

Step 2

Review the S3 Publisher Plugin

One example plugin is an S3 publisher plugin. In a real CI/CD workflow, Jenkins should not treat the workspace as permanent artifact storage. A plugin can help move build outputs to a better storage location such as AWS S3.

Step 3

Copy the Previous Jenkins Project

To keep the lab focused, I copied the existing Maven Jenkins project instead of creating every setting again from zero. That made it easier to concentrate on artifact naming and plugin behavior.

Step 4

Use BUILD_ID for Automatic Versioning

Jenkins provides built-in environment variables. One useful variable is $BUILD_ID, which changes from build to build. I used it to create a WAR file name that does not overwrite the previous output.

mvn install mkdir -p versions cp target/vprofile-v2.war versions/vpro-$BUILD_ID.war

The mkdir -p command creates the versions directory if it does not already exist. The final command copies the Maven WAR output into that folder with a build-specific file name.

Step 5

Install the Build Timestamp Plugin

Another option is timestamp-based versioning. After installing the Build Timestamp plugin, Jenkins can expose a timestamp value through $BUILD_TIMESTAMP. This is useful when the artifact name should show when the build happened.

mvn install mkdir -p versions cp target/vprofile-v2.war versions/vpro-$BUILD_TIMESTAMP.war
Step 6

Enable Timestamp Variables in System Settings

After installing a plugin, Jenkins may need extra configuration before the new variable is available in a job. For the timestamp plugin, the related settings are enabled from Jenkins system configuration.

Step 7

Create a Manual VERSION Parameter

Jenkins also supports parameterized builds. I enabled This project is parameterized and added a string parameter named VERSION. This lets me enter a version number before starting the build.

mvn install mkdir -p versions cp target/vprofile-v2.war versions/vpro-$VERSION.war
Step 8

Build with a Custom Version

After the parameter is enabled, Jenkins asks for the version value before running the job. For example, entering 2.3.6 creates an artifact named vpro-2.3.6.war.

Step 9

Confirm the Versioned WAR File

The final check is to inspect the workspace and confirm the WAR file exists in the versions folder. This proves that Jenkins did more than run Maven: it created a traceable build output.

Where Artifacts Should Go Next

The Jenkins workspace is not the final place to store production artifacts. It is useful during the build, but real release assets should be stored somewhere durable and searchable.

AWS S3

A simple object storage option for storing versioned build files and deployment packages.

Nexus Repository

A repository manager commonly used to store build artifacts, packages, and dependency assets.

JFrog Artifactory

A dedicated artifact platform for managing binary releases and deployment-ready outputs.

My main takeaway from this lab is that artifact versioning is a small shell-script change with a big CI/CD impact. It turns a basic build output into something that can be tracked, compared, deployed, and rolled back.
