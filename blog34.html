Understanding Jenkins, Nexus and SonarQube in a CI Pipeline

My practical learning from setting up Nexus and SonarQube with Jenkins on AWS EC2, including what each tool does and how the network access fits together.

GitHub -> Jenkins -> Maven Build -> SonarQube Analysis -> Nexus Artifact Repository
DevOps Jenkins Nexus SonarQube CI/CD AWS EC2

Introduction

In this blog, I am documenting my understanding of how Jenkins, Nexus, and SonarQube work together in a Continuous Integration pipeline. At first, I was confused about the role of Nexus and SonarQube. I was thinking Nexus might be something like Docker. After learning the flow, I now understand that Jenkins, Nexus, and SonarQube each have different responsibilities in the pipeline.

Jenkins is the automation tool, SonarQube checks the quality and security of the code, and Nexus stores the final build artifacts such as .war or .jar files.

Simple idea: Jenkins builds the application, SonarQube checks the code quality, and Nexus stores the final build package.

Basic CI Pipeline Flow

Developer pushes code to GitHub ↓ Jenkins pulls the code ↓ Jenkins builds the code using Maven ↓ Jenkins sends code to SonarQube for quality analysis ↓ Jenkins uploads the final artifact to Nexus ↓ Artifact can be used later for deployment

This flow helps teams automate the build process, check code quality early, and store build versions properly for future deployment or rollback.

What is Jenkins?

Jenkins is an automation server. In a DevOps pipeline, Jenkins controls the steps of the CI/CD process. It can pull source code from GitHub, run Maven commands, execute tests, send code to SonarQube, and upload artifacts to Nexus.

I like to think of Jenkins as the manager of the pipeline. It does not mainly store artifacts and it does not mainly analyze code quality by itself. Instead, it connects with other tools and runs the full process.

Jenkins responsibilities

  • Pull code from GitHub.
  • Build the application using Maven.
  • Run tests and pipeline stages.
  • Send code to SonarQube for analysis.
  • Upload build artifacts to Nexus.

Pipeline mindset

Jenkins coordinates the work, but the specialist tools do the deeper jobs: SonarQube analyzes quality and Nexus keeps the release package.

What is Nexus?

Nexus is an artifact repository. It stores final build files created by Jenkins. For example, in a Java web application, Maven may generate a .war file after the build. Jenkins can upload this file to Nexus.

Nexus is not exactly like Docker. A better comparison is that Nexus is like Docker Hub, AWS ECR, or GitHub Packages because it stores built packages or images. Docker is used to build and run containers, while Nexus is used to store artifacts.

vprofile-1.0.war vprofile-1.1.war vprofile-1.2.war

This is useful because if a new version has a problem, we can go back and deploy an older version from Nexus.

Why do we need Nexus?

  • To store final build artifacts.
  • To keep different versions of application packages.
  • To make rollback easier.
  • To share build artifacts with deployment tools.
  • To avoid depending only on the Jenkins workspace.
My understanding: GitHub stores source code versions. Nexus stores final build package versions.

What is SonarQube?

SonarQube is a code quality and security analysis tool. It checks the source code and reports issues such as bugs, vulnerabilities, code smells, duplicated code, and test coverage.

A project can compile successfully but still have poor-quality code. SonarQube helps detect these problems before the application is deployed.

Bugs: 3 Security Vulnerabilities: 1 Code Smells: 15 Duplicated Code: 5% Test Coverage: 68%

Why do we need SonarQube?

  • To detect bugs early.
  • To find security vulnerabilities.
  • To reduce duplicate code.
  • To improve maintainability.
  • To enforce quality gates before deployment.

Difference Between GitHub, Jenkins, Nexus, Docker Registry and SonarQube

Tool Main Purpose Stores What?
GitHub Source code repository Code and commits
Jenkins Automation server Pipeline jobs and build history
Nexus Artifact repository .jar, .war, npm packages, Maven artifacts, Docker images
Docker Registry Container image repository Docker images
SonarQube Code quality analysis Code analysis reports

How Jenkins Connects With Nexus

Jenkins connects to Nexus to upload the final build artifact. In this project, Nexus runs on port 8081. That means the Nexus security group must allow Jenkins to access port 8081.

Jenkins -> Nexus Port -> 8081 Purpose -> Upload build artifact

Example: Jenkins builds the vProfile project and creates a WAR file. Then Jenkins uploads that WAR file to Nexus so that it can be used later for deployment.

How Jenkins Connects With SonarQube

Jenkins connects to SonarQube to send the code for analysis. In this setup, SonarQube is behind NGINX, so Jenkins connects to SonarQube using port 80.

Jenkins -> SonarQube Port -> 80 Purpose -> Send code for quality analysis

SonarQube also needs to send the quality gate result back to Jenkins. For this, Jenkins must allow traffic from the SonarQube security group on port 8080.

SonarQube -> Jenkins Port -> 8080 Purpose -> Send quality gate result back to Jenkins

My Simple Analogy

To understand the flow better, I used a simple assignment submission analogy.

Jenkins = Person managing the whole submission process SonarQube = Teacher checking quality and mistakes Nexus = File cabinet storing final submitted versions

Jenkins takes the code, builds it, asks SonarQube to check the quality, and stores the final output in Nexus.

Final Summary

GitHub = Stores source code Jenkins = Runs the automation pipeline SonarQube = Checks code quality and security Nexus = Stores final build artifacts

This lecture helped me understand why real DevOps pipelines use multiple tools instead of doing everything inside Jenkins. Jenkins controls the process, but specialized tools like SonarQube and Nexus make the pipeline more reliable, maintainable, and production-ready.

Key takeaway: Jenkins builds and controls the pipeline, SonarQube improves code quality, and Nexus stores versioned build artifacts for future deployment.
