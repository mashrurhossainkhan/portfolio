Fixing Jenkins Disk Space Issue on AWS EC2

This lab covers a common Jenkins problem: running out of disk space on a small EC2 root volume. I practiced checking Linux disk usage, increasing the AWS EBS volume, and confirming the new filesystem space.

Jenkins Error → Linux Disk Check → EBS Volume Resize → Instance Reboot → Filesystem Verification

Jenkins creates and stores many files while it runs: plugins, build logs, job workspaces, Maven dependencies, temporary files, archived artifacts, and build history. If Jenkins is running on an AWS EC2 instance with a small root volume, storage can fill up quickly.

Main lesson: Jenkins is not only a CI/CD application. It is also a server that needs normal operations care, including disk monitoring, cleanup, and enough storage for build activity.

The Problem

In this setup, Jenkins was running on an EC2 instance with the default 8 GB EBS root volume. After installing plugins and running several jobs, Jenkins can fail with an error like this:

No space left on device

This means the Linux filesystem does not have enough available space to continue writing files. For Jenkins, that can stop builds, plugin installs, log writing, dependency downloads, or artifact creation.

What Usually Consumes Jenkins Disk Space

Workspaces

Each job can leave checked-out source code, generated files, build outputs, and temporary folders.

Build history

Jenkins keeps console logs, build metadata, and archived outputs unless cleanup policies are configured.

Dependencies

Maven, npm, Docker, and other tools may cache dependencies on the Jenkins server.

Plugins

Installed Jenkins plugins and their supporting files also consume space under the Jenkins home directory.

Step 1

Check the Current Disk Usage

First, I logged in to the Jenkins EC2 server and switched to the root user so I could inspect system storage clearly.

sudo -i df -h

The df -h command shows filesystem usage in a human-readable format. It shows the total size, used space, available space, mounted path, and usage percentage.

Step 2

Check the Disk and Partition Layout

Next, I checked the disk and partition information. This helped me understand whether the full disk was small or whether the partition/filesystem still needed to be expanded.

fdisk -l
One important detail: /dev/xvda represents the full disk, while /dev/xvda1 represents a partition inside that disk. The disk size and the usable filesystem size may not always match until the partition and filesystem are resized.
Step 3

Increase the AWS EBS Volume Size

To fix the storage problem, I increased the Jenkins EC2 root volume from 8 GB to 20 GB from the AWS Console.

EC2 → Volumes → Select Jenkins Root Volume → Actions → Modify Volume 8 GB → 20 GB

This increases the EBS storage attached to the instance. It is the cloud storage layer, so the operating system still needs to detect and use the new size afterward.

Step 4

Reboot the Jenkins EC2 Instance

After modifying the volume, I rebooted the Jenkins EC2 instance so the operating system could detect the updated storage size cleanly.

EC2 → Instance State → Reboot Instance
Step 5

Verify the New Disk Size

Once the instance came back online, I logged in again and checked the disk layout and filesystem usage.

sudo -i fdisk -l df -h

After the resize, the disk showed around 20 GB. The root filesystem showed around 19 GB of usable space, which is normal because some space is reserved for system usage and filesystem metadata.

Useful Cleanup Commands

Increasing the volume fixes the immediate storage limit, but cleanup is still useful. Old packages, logs, images, and containers can also consume a lot of space.

sudo apt clean sudo journalctl --vacuum-time=7d docker system prune -a
Be careful with docker system prune -a. It removes unused Docker images and containers, so it should only be run when you understand what can be safely deleted.

What I Learned

  • Jenkins can quickly run out of disk space during CI/CD work.
  • df -h shows filesystem usage clearly.
  • fdisk -l helps inspect disks and partitions.
  • AWS EBS volumes can be resized from the EC2 console.
  • Build servers need storage monitoring, cleanup, and retention policies.
Increasing the EBS root volume from 8 GB to 20 GB is a simple fix, but the bigger DevOps lesson is to monitor Jenkins storage before jobs fail. CI/CD servers are production tools, and their disk usage needs the same attention as application servers.
